## Хеш-таблица

### Устройство

Внутри — массив указателей (`node**`), каждый указатель — голова списка, а каждый `node` содержит key, value и next.
Это называется **метод цепочек**: при коллизии элементы не вытесняют друг друга, а образуют цепочку в одном бакете, колизионный ключ добавляется в нужный бакет в поле next элемента, уже хранящегося в этом бакете.

```
buckets[0] --> NULL
buckets[1] --> ["banana"] --> ["cherry"] --> NULL   // коллизия
buckets[2] --> ["apple"] --> NULL
```

Выбрал метод цепочек — он оказался мне ближе ближе по духу и проще по реалаизации - при удалении элемента достаточно просто вытащить узел из списка и освободить память. При открытой адресации удалённую ячейку нельзя просто обнулить: поиск идёт по цепочке смещений, и пустая ячейка означает «дальше не ищи». Поэтому удалённые ячейки помечаются статусом DELETED — они не занятые, но и не пустые.

### Хеш-функция — FNV-1a
64-битная хэш-функция FNV-1a-наследник FNV1

```c
#define FNV1A64_OFFSET_BASIS  14695981039346656037u
#define FNV1A64_PRIME         1099511628211u

static size_t fnv1a_hash(const char* key, size_t capacity)
{
    size_t index = FNV1A64_OFFSET_BASIS;
    while (*key) {
        index = index ^ (unsigned char)*key++;
        index = index * FNV1A64_PRIME;
    }
    return index % capacity;
}
```

Алгоритм следующий: берём каждый байт строки, делаем XOR с текущим значением хеша, умножаем на простое число. Повторяем до конца строки.

Выбрал FNV-1a за:
- хороший лавинный эффект - малые изменения на входе - большая разница в хэше
- простоту в реализации
- не криптографическая (в данном случае прятать нечего)
- частое попадание в поле зрения

Еще читал про и djb2, sdbm - как я понял, константы в них подбирались на глазок, а у FNV есть обоснование на теории вероятностей.
Сравнение хеш-функций для строк: [wiki.rakovets.by — Hashing](https://wiki.rakovets.by/v0/hashing/)

### Рехеширование

Когда таблица заполняется больше чем на 75%, она автоматически удваивается.  
Порог 0.75 — стандартный компромисс: и память не тратим впустую, и цепочки не вырастают длинными.

При рехешировании узлы **не пересоздаются** — они физически "переезжают" в новый массив.

## Множество (Set)

Set — обёртка над хеш-таблицей. Сама таблица дубликаты не запрещает, проверка уникальности живёт в set_add(), и если элемент уже есть, вставка не происходит. Поле value в узле таблицы для set не несёт смысла — может все таки стоило оставить node без value, а для multiset сделать отдельный счетчик.
Реализованы операции:
- `set_union` — объединение (все элементы из обоих множеств)
- `set_intersection` — пересечение (только общие элементы)
- `set_difference` — разность (есть в первом, нет во втором)

Теория про множества: [neerc.ifmo.ru — Множества](https://neerc.ifmo.ru/wiki/index.php?title=%D0%9C%D0%BD%D0%BE%D0%B6%D0%B5%D1%81%D1%82%D0%B2%D0%B0)

## Мультимножество (Multiset)

Multiset отличается от set тем, что один элемент может встречаться несколько раз.
Multiset тоже построен на хеш-таблице key → void*, но здесь void* используется осмысленно — туда кладётся указатель на счётчик. При добавлении элемента multiset_add сначала ищет его в таблице: если нашёл — увеличивает счётчик, если нет — создаёт новый узел со счётчиком = 1.

Реализованы операции:
- `multiset_sum` — сумма (вхождения складываются)
- `multiset_union` — объединение (берётся максимум вхождений)
- `multiset_intersection` — пересечение (берётся минимум вхождений)
